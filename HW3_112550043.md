## Experimental Setup

This homework is a multimodal retrieval task. For each question, the system must
return one combined top-5 list of `quote_id`s from both `text_quotes` and
`img_quotes`. The evaluation metric is `Recall@5`.

An important property of `Recall@5` is that the internal order of the top 5
does not matter. Rank 1 and rank 5 receive the same credit as long as the gold
evidence appears somewhere in the top-5 set. Therefore, the real objective is
top-5 evidence coverage rather than fine-grained top-1 ranking.

Dataset statistics:

| Split | Samples | Avg. candidates | Avg. text candidates | Avg. image candidates |
|---|---:|---:|---:|---:|
| Train | 2055 | 14.997 | 10.000 | 4.997 |
| Test | 1798 | 14.979 | 9.989 | 4.990 |

I used multiple validation protocols during the project:

1. `document-level split`:
   early baseline evaluation, useful for fast comparison but optimistic
2. `row-level split`:
   more random, but still optimistic because it keeps many repeated patterns
3. `mixed split`:
   a stronger proxy that preserves some overlap structure while reducing leakage
4. `domain holdout split`:
   evaluates cross-domain robustness by holding out one domain such as
   `Academic paper`

The reason for moving beyond a single split is that local validation and Kaggle
public score diverged strongly. Some local splits reached `0.86+`, while the
public leaderboard stayed around `0.80`. Later analysis showed that the main
reason was domain shift.

### Kaggle summary

- `submission_gate_seen_docheavy3.csv`

## Q1. Method Description

### Overall retrieval pipeline

My method treats the task as a per-question ranking problem over a small
candidate pool. For each sample:

1. Convert all text and image evidence into a shared text representation.
2. Score every candidate with multiple complementary retrievers.
3. Add train-set memory signals when the document has useful overlap with
   training data.
4. Use a supervised ranker to combine these features.
5. Optionally rerank the top candidates with a cross-encoder.
6. Output the final top 5 `quote_id`s.

The final high-performing family is not a single simple retriever. It is a
doc-aware ranking pipeline with memory features, reranking, and lightweight
submission-level gating.

### Evidence preprocessing

All candidates are converted into text before retrieval.

- For `text_quotes`, I use the original `text` field.
- For `img_quotes`, I use the provided `img_description` field.

This is effective because many "images" in this dataset are actually:

- tables
- charts
- figures
- document screenshots

Their descriptions already contain the useful semantic content, such as numbers,
labels, trends, and table summaries.

Before retrieval, I normalize the text by:

- stripping surrounding spaces
- replacing newlines and tabs with spaces
- normalizing repeated whitespace
- applying lightweight tokenization for BM25

I also prefix each candidate with a type marker such as `[TEXT]`, `[TABLE]`,
`[FIGURE]`, or `[IMAGE]`. This keeps all evidence in one text space while
exposing a small amount of modality/type information.

### Retriever components

I implemented and compared the following retrieval components:

1. `BM25`
   lexical matching over question and evidence text
2. `Dense Retriever`
   sentence-embedding similarity using `BAAI/bge-large-en-v1.5`
3. `Direct LLM Selection`
   prompting `Qwen/Qwen2.5-7B-Instruct` to choose 5 `quote_id`s directly
4. `Memory-based retrieval`
   train gold evidence memory indexed by document/content/page/layout/type
5. `Same-document similar-question memory`
   retrieve train questions from the same document and let their gold evidence
   vote for current candidates
6. `Cross-encoder reranker`
   mainly `mixedbread-ai/mxbai-rerank-large-v1`
7. `Dual reranker ensemble`
   combine two rerankers and the base ranking with reciprocal rank fusion

### Ranking procedure

I used three increasingly strong ranking procedures.

#### 1. Hybrid score fusion

The early strong baseline was:

```text
hybrid_score = 0.35 * normalized_bm25 + 0.65 * normalized_dense
```

This already outperformed BM25, dense retrieval, and direct LLM selection.

#### 2. Memory-enhanced hybrid ranking

I then extended the score with:

- train gold memory score
- same-document similar-question memory score
- expected modality/type prior
- page context boost
- layout context boost

This improved public score because many test questions come from documents that
also appear in train.

#### 3. Doc-aware supervised ranker

The strongest single-model architecture was `doc_aware_ranker`.

For each candidate, I extract features such as:

- normalized BM25 score
- normalized dense score
- memory score
- precise memory score
- similar-memory score
- page and layout context scores
- neighbor votes from similar train questions
- document overlap confidence
- candidate modality and raw type
- token length and rank percentiles

These features are fed into a `HistGradientBoostingClassifier`, which predicts a
candidate relevance score. I then combine the learned score with the original
base score and optionally rerank the top candidates with `mxbai`.

The best single-model configuration was:

- method: `doc_aware_ranker`
- `bm25_weight=0.2`
- `dense_weight=0.65`
- `memory_weight=0.55`
- `similar_memory_weight=0.15`
- `type_weight=0.0`
- `ranker_model_weight=0.8`
- `ranker_base_weight=0.2`
- reranker: `mixedbread-ai/mxbai-rerank-large-v1`
- `reranker_weight=0.45`

### Additional techniques

I also tried several extra techniques:

- `page/layout context boost`:
  reward candidates near strong evidence on the same page or layout
- `modality/type prior`:
  slightly boost the modality type expected by the question metadata
- `precise doc memory`:
  exact document-local memory keys
- `dual reranker RRF`:
  fuse two rerankers and the base ranking
- `submission-level RRF`:
  combine outputs from several high-performing systems
- `seen-document gating`:
  only apply memory-heavy or doc-heavy fusion when `doc_name` was seen in train
- `question-type routing`:
  use one prediction set for `Descriptive` questions and another for
  `non-Descriptive` questions

The key design lesson was that the best improvements came from better top-5
coverage and better routing of overlap signals, not from making rank 1 more
accurate.

### Analysis of the high-public-score methods

#### 1. Memory hybrid

`memory_hybrid_rerank` was the first major jump over the baseline. It combined:

- BM25
- dense retrieval
- gold evidence memory
- similar-question memory
- reranking

This improved public score because test/train document overlap is real, and the
image descriptions are informative enough to benefit from both lexical and
semantic retrieval.

#### 2. Dual reranker

`dual_reranker` improved from `0.76887` to `0.77041` by ensembling two
rerankers plus the base ranking with RRF. This helped top-5 coverage, but the
gain was small because rerankers cannot fully correct mistakes caused by
overconfident memory signals.

#### 3. Doc-aware supervised ranker

`docaware_tune1` was the largest model-side improvement. It jumped to `0.79969`
because it learned when to trust:

- semantic signals
- lexical signals
- document overlap
- precise memory
- neighbor votes from similar train questions

This was stronger than any fixed linear fusion.

#### 4. Submission-level RRF

`rrf_docheavy3` improved slightly further to `0.80200`. Its effect came from
coverage. Different systems retrieved different true evidence items, and RRF
preserved more of that union in the top 5.

#### 5. Seen-document gating

`gate_seen_docheavy3` produced the best verified public score `0.80277`. This
method only applied doc-heavy fusion to test rows whose `doc_name` also existed
in train. That routing decision was important because overlap signals are useful
for seen documents but can hurt on out-of-domain examples.

#### 6. Question-type routing

The later `gate_nondescriptive_sim15` system tied the best public score. It
kept the best public submission as the base and only replaced predictions on
`non-Descriptive` questions with the stronger `sim15` single-model output. This
was the best local routing configuration, but it did not exceed the previous
best public score.

### Validation shift analysis

Later experiments showed that local validation was still too optimistic.

The baseline doc-aware model achieved:

This drop is very close to the Kaggle public range. Therefore, the main problem

These unseen domains account for about `22.7%` of the test set.

This explains why local scores above `0.86` did not translate into public scores

## Q3. Multimodal Embedding vs. Text-Description Retrieval for Image Evidence

I chose option `(b)`: convert each image into text and retrieve it together
with text evidence using the provided `img_description` field.

I also implemented a raw-image baseline using `openai/clip-vit-base-patch32`.
In that baseline, image candidates were encoded directly from pixels and
compared with the question embedding.

Validation comparison:

| Image evidence approach | Method | Recall@5 |
|---|---|---:|
| Raw image embedding | CLIP baseline | 0.585678 |
| Text-description retrieval | dense retriever over `img_description` | 0.771848 |
| Text-description retrieval | hybrid retriever | 0.804594 |
| Text-description retrieval | doc-aware ranker family | 0.867378 to 0.875634 on mixed split |

Text-description retrieval was much stronger than direct image embedding.

The reason is that this dataset is closer to document/table understanding than
to natural-image retrieval. Many image candidates are:

- tables
- charts
- document figures
- slides

The useful evidence is often exact textual or numeric content. The provided
`img_description` fields expose that information directly in text, which makes
both BM25 and dense retrieval effective. In contrast, CLIP is better at broad
image-text alignment than at matching fine-grained OCR-like table details.

I therefore used text-description retrieval throughout the final system.

These numbers show that the final system is almost perfectly balanced between
text and image evidence.

This is notable because the candidate pool itself is not balanced: each sample
has about 10 text candidates but only about 5 image candidates. In other words,
the model does not merely follow candidate frequency. It promotes image
candidates aggressively when their descriptions contain useful evidence.

### Why the system stays balanced

The system stays balanced for two reasons:

1. text quotes and image descriptions are scored in the same retrieval space
2. many image candidates are actually tables, which contain exactly the kind of
   structured evidence that financial and comparative questions need

This also matches the error analysis. On the `Academic paper` holdout, the main
remaining weakness was not `image_only` questions:

- `image_only`: `0.933908`
- `mixed`: `0.755902`

Therefore, the bottleneck is not a failure to retrieve image evidence. The
harder cases are mixed-modality questions that require broader top-5 coverage.
