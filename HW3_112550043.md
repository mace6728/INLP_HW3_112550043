## Q1. Method Description

My final method is a hybrid retriever that ranks all evidence candidates within
each question's own candidate pool. Each sample contains approximately 10 text
candidates and 5 image candidates. The goal is not to retrieve five text items
and five image items separately, but to return one combined top-5 list of
`quote_id`s across all modalities.

### Evidence preprocessing

All candidates are converted into text before retrieval. For normal text
evidence, I use the original `text` field. For image, chart, figure, and table
evidence, I use the provided `img_description` field. This is important because
many image candidates are actually tables or charts whose descriptions contain
numbers, labels, and relationships that are directly relevant to the question.

Before scoring, text is normalized by:

- stripping leading and trailing spaces
- replacing newlines, tabs, and non-breaking spaces with regular spaces
- removing or normalizing some LaTeX-like symbols
- collapsing repeated whitespace
- tokenizing alphanumeric terms for BM25

Each candidate is also prefixed with a modality/type marker such as `[TEXT]`,
`[TABLE]`, `[FIGURE]`, or `[IMAGE]`. This gives the dense model and BM25 scorer
a small amount of type information while still keeping all evidence in a shared
text space.

### Retrievers

I implemented and compared four retrieval styles:

| Method | Description |
|---|---|
| BM25 | Lexical retrieval over normalized question and candidate text. |
| Dense Retriever | Encodes questions and candidate text with `BAAI/bge-large-en-v1.5`, then ranks by cosine similarity. |
| Direct LLM Selection | Prompts `Qwen/Qwen2.5-7B-Instruct` to directly output the five most relevant `quote_id`s. |
| Hybrid Retriever | Combines min-max normalized BM25 and dense scores. |

The final submitted method is the hybrid retriever. For each sample, I compute:

```text
hybrid_score = 0.35 * normalized_bm25_score + 0.65 * normalized_dense_score
```

Then I sort all candidates by this score and output the top five `quote_id`s.

### Ranking procedure

The ranking is performed independently for each question. No global index is
needed because each question already comes with its own candidate evidence pool.
For each sample:

1. Convert all `text_quotes` and `img_quotes` into candidate text.
2. Compute BM25 scores between the question and all candidates.
3. Compute dense embedding similarity scores between the question and all candidates.
4. Min-max normalize both score arrays within the same sample.
5. Fuse the two normalized score arrays with weights 0.35 and 0.65.
6. Return the top five candidate `quote_id`s after stable score sorting.

I did not use closed-source APIs. All models are open-weight and below the 80B
parameter limit.

## Q2. Comparison of Retrieval Methods

Validation results:

| Method | Model / Approach | Recall@5 |
|---|---|---:|
| BM25 | Lexical matching | 0.764501 |
| Dense Retriever | `BAAI/bge-large-en-v1.5` | 0.771848 |
| Direct LLM Selection | `Qwen/Qwen2.5-7B-Instruct` | 0.726621 |
| Final Hybrid Method | BM25 + dense fusion | 0.804594 |

### BM25

BM25 is a strong baseline for this task. Its Recall@5 was `0.764501`, which is
only slightly below dense retrieval. This shows that many questions contain
keywords, entities, dates, metric names, or table labels that also appear in the
evidence descriptions. BM25 is especially effective when the question asks for
specific financial terms, named fields, or exact phrases.

The weakness of BM25 is that it is lexical. If the evidence uses paraphrased
language or if the relevant image/table description does not share many words
with the question, BM25 may rank it too low. It also has limited ability to
understand that two semantically related phrases should match when their surface
tokens differ.

### Dense Retriever

The dense retriever achieved `0.771848`, slightly better than BM25. This method
uses `BAAI/bge-large-en-v1.5` to encode the question and every candidate text
description into a shared embedding space. It is better than BM25 when the
question and the evidence are semantically related but do not share exact
tokens.

However, dense retrieval is not always better than lexical matching. Some
questions require exact numeric or tabular evidence, and dense embeddings can
blur fine-grained details such as year, metric name, or column label. For these
cases, BM25's exact token matching remains useful.

### Direct LLM Selection

The direct LLM selection baseline achieved `0.726621`, lower than both BM25 and
dense retrieval. I prompted `Qwen/Qwen2.5-7B-Instruct` with the question and the
candidate list, asking it to return a JSON array of five `quote_id`s.

The LLM can reason about candidate relevance, but this approach has several
weaknesses:

- the prompt must include many long candidates, so evidence may be truncated
- the model can focus on plausible-looking evidence instead of exact support
- output parsing can fail or require fallback behavior
- inference is much slower than scoring-based retrieval

Because the task only requires selecting evidence from a small candidate pool,
scoring-based retrieval is more stable and efficient than asking a generative
model to perform the whole selection.

### Final Hybrid Method

The hybrid method achieved the best Recall@5: `0.804594`. It improves over BM25
by `0.040093` and over dense retrieval by `0.032746`.

The main reason is complementarity. BM25 captures exact matches for entities,
numbers, years, and financial/table labels. Dense retrieval captures semantic
similarity when the wording differs. Combining them reduces the chance that a
candidate is missed because it is weak under only one scoring method.

I selected the hybrid method as the final submission method because it had the
best validation Recall@5 and produced the Kaggle submission with public score
`0.71571`.

## Q3. Multimodal Embedding vs. Text-Description Retrieval for Image Evidence

I mainly chose text-description retrieval for image evidence. That means I used
the provided `img_description` field and ranked image candidates together with
text candidates using BM25, dense retrieval, and the final hybrid method.

I also ran a raw-image multimodal embedding baseline with `openai/clip-vit-base-patch32`.
In this baseline, image candidates are encoded directly from image pixels with
CLIP, while text candidates are still handled with BM25. The question is encoded
with CLIP's text encoder and compared with image embeddings.

Validation comparison:

| Image Evidence Approach | Method | Recall@5 |
|---|---|---:|
| Raw image embedding | CLIP image retrieval baseline | 0.585678 |
| Text-description retrieval | Dense text retriever over `img_description` | 0.771848 |
| Text-description retrieval | Final hybrid method | 0.804594 |

Text-description retrieval performed much better than raw-image embedding on
this task. The gap between CLIP raw-image retrieval and the final hybrid method
was large: `0.585678` vs. `0.804594`.

The likely reason is that many image candidates are tables, charts, or document
figures. The relevant information is often exact text, numbers, column names,
and relationships inside the image. CLIP is good at broad image-text alignment,
but it is not optimized for reading detailed table values or matching financial
metrics from document screenshots. In contrast, the provided `img_description`
already converts these visual elements into natural language, including many
important numbers and labels. Once image evidence is represented as text, both
BM25 and BGE can compare it to the question more directly.

Therefore, for this dataset, converting image evidence into text descriptions is
more effective than directly embedding raw images.

## Q4. Modality Preference Analysis

I analyzed the top-5 retrieval results of the final hybrid method on the same
validation split.

Overall modality counts in the top-5 results:

| Modality | Count | Ratio |
|---|---:|---:|
| Text | 620 | 0.5439 |
| Image | 520 | 0.4561 |

Average modality composition per top-5 list:

| Metric | Value |
|---|---:|
| Avg. text candidates in top-5 | 2.7193 |
| Avg. image candidates in top-5 | 2.2807 |

Raw candidate type counts:

| Raw type | Count |
|---|---:|
| text | 620 |
| table | 482 |
| image | 38 |

The final retriever has a mild preference for text evidence, but it is not
strongly biased. About 54.39% of selected top-5 evidence items are text
candidates, while 45.61% are image candidates. This is close to balanced,
especially considering that each sample usually contains around 10 text
candidates and only around 5 image candidates.

This behavior comes from the design of the hybrid method. Both text candidates
and image candidates are converted into the same textual representation space.
Image candidates are not handled as a separate retrieval branch; their
`img_description` fields compete directly with text quotes in the same top-5
ranking. As a result, an image/table candidate can be ranked highly whenever its
description matches the question well.

The slight preference for text evidence is still expected. Text quotes are more
numerous in the candidate pool, and BM25 can strongly reward exact word overlap
in textual passages. Some image descriptions are summaries rather than complete
OCR outputs, so they may omit exact values or labels needed by the question.
This can make text evidence more competitive in some cases.

The most common selected image raw type was `table`, with 482 selected table
candidates. This suggests that the model frequently relies on table
descriptions, which matches the dataset's evidence distribution: many questions
ask about financial values, comparisons, or structured information. The final
method therefore does not ignore image evidence; it mainly favors image
candidates when their textual descriptions contain useful structured evidence.

## Conclusion

The best method in my experiments was the hybrid retriever, which combines BM25
and dense text retrieval over both text evidence and image descriptions. It
achieved the highest validation Recall@5 (`0.804594`) and produced the submitted
Kaggle file. The experiments show that exact lexical matching and semantic
embedding retrieval are complementary, and that image descriptions are much more
useful than raw image embeddings for this document evidence retrieval task.
